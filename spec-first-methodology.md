## 01 Философия: почему Spec-First

Большинство разработчиков начинают с кода. Открывают IDE, создают проект, пишут первый компонент и через неделю понимают, что архитектура не выдерживает. Al-driven разработка усиливает эту проблему: если агент не понимает контекст проекта целиком, он генерирует фрагментарный код, который не складывается в систему.

**Spec-First** это принцип, при котором ни одна строка кода не пишется, пока проект полностью не описан в документации.

Спецификация это не бюрократия. Это инструмент управления Al-агентами. Когда Claude Code получает детальную спецификацию, он работает автономно: понимает связи между модулями, выбирает правильные паттерны, не задаёт лишних вопросов.

| Без спецификации | Со спецификацией |
| :--- | :--- |
| Claude спрашивает на каждом шаге | Claude работает автономно |
| Конфликты между модулями | Модули стыкуются по архитектуре |
| Переписывание через 3 дня | Итерации - улучшение, не переделка |
| Ручной контроль каждой строки | Субагенты работают параллельно |
| Дни на настройку окружения | 5 минут: скопировал промпт поехал |

**Ключевой принцип**
Спецификация это ТЗ для Al-агентов, а не для людей. Чем конкретнее описаны таблицы, эндпоинты, компоненты тем автономнее работает Claude Code.

## 02 Пайплайн: 5 слоёв от идеи до деплоя

Методология строится на 5 последовательных слоях. Каждый слой трансформирует данные предыдущего слоя в более конкретный формат.

* **01 Документ идеи**: Проблема, решение, аудитория, архитектура, монетизация, конкуренты, план запуска. Аналитический документ, из которого Al извлечёт все параметры.
* **02 Техническая спецификация**: Детальный чертёж модулей: таблицы с типами полей, АРІ-эндпоинты, user stories, UI-экраны, крайние случаи. Формат для реализации без уточнений.
* **03 Генератор конфигурации**: Системный промпт, превращающий спецификацию в набор файлов: CLAUDE.md, субагенты, rules, skills, МСР-серверы.
* **04 Пакет конфигурации**: Готовые файлы: CLAUDE.md (до 120 строк), 4-6 субагентов, rules, skills, шаблон фичи, МСР-команды. Копируется одним промптом.
* **05 Автономная сборка**: Claude Code c субагентами реализует проект. database-architect, backend-engineer, frontend-developer, qa-reviewer работают параллельно.

## 03 Слой 1 — Документ идеи проекта

`PROJECT_IDEA.md` первый и самый важный документ. Он пишется для Al, который будет анализировать проект и генерировать спецификацию.

**Обязательные разделы**
1.  **Проблема**: Конкретная боль с числами. Не «людям неудобно» а «разработчик тратит 3-5 часов вместо 5 минут».
2.  **Решение**: Пошаговый процесс: «принимает идею выдаёт». Каждый шаг станет модулем. анализирует рекомендует генерирует →
3.  **Почему сейчас**: Тайминг рынка: технологии, ниши, конкурентная среда.
4.  **Целевая аудитория**: Основная и вторичная. Кто, какую задачу решают, текущий инструментарий.
5.  **Архитектура**: Слои системы с диаграммой. Стек с обоснованием. Модули с входами/выходами.
6.  **Монетизация**: Планы, цены, гейтинг фич. Al использует для генерации таблиц подписок.
7.  **Конкуренты**: Таблица: конкурент что делает чего не хватает.
8.  **План запуска**: Фазы с целями и метриками. МѴР выделен отдельно.
9.  **Риски**: Таблица: риск вероятность митигация.
10. **Техдетали**: Структура репозитория, ключевые таблицы, АІ-пайплайн.

**Правило конкретики**
Вместо «продвинутая аналитика» пиши: «дашборд с графиками retention по когортам, данные из Supabase через Edge Function cron». Al не работает с абстракциями. конверсия free pro, WAU/MAU

## 04 Слой 2 — Техническая спецификация

Спецификация трансформация документа идеи в формат для автоматической генерации кода. Каждый модуль описан так, что субагент Claude Code может реализовать его без уточнений.

**6 обязательных блоков каждого модуля**
1.  **User Stories**: «Как [роль , я хочу действие), чтобы [результат]». Минимум 3-5 историй. Основной сценарий и крайние случаи.
2.  **Модель данных**: Таблицы: имя, тип (uuid, text, jsonb, timestamptz), constraints, FK-связи, RLS-политики. Готово к SQL.
3.  **АРІ и серверная логика**: Эндпоинты: метод, путь, тело, ответ, коды ошибок. Или Server Actions c входами и результатом.
4.  **Экраны и компоненты**: Страницы и компоненты: что отображают, действия, состояния (загрузка, ошибка, пусто, успех).
5.  **Бизнес-логика**: Правила валидации, ограничения, формулы, условия переходов между состояниями.
6.  **Крайние случаи**: Нет данных? АРІ не отвечает? Конкурентное редактирование? Конкретные сценарии с ожидаемым поведением.

**Пример: модуль аутентификации**
* User Stories:
    * Как новый пользователь, я хочу зарегистрироваться через email
    * Как пользователь GitHub, я хочу войти через OAuth
* Модель данных:
    * users: id (uuid PK), email (text UNIQUE NOT NULL), full_name (text), plan (text DEFAULT 'free'), created_at (timestamptz)
    * RLS: SELECT: auth.uid() = id | UPDATE: auth.uid( $)=id$
* API:
    * POST/auth/signup email, password) → 201 | 400 | 409
    * POST/auth/login email, password → 200 + session | 401

**Формат данных**
Типы - как в SQL: uuid, text, jsonb, timestamptz. Эндпоинты как в коде: POST /api/projects. Чем ближе к коду тем точнее генерация.

**Промпт-инструкция для генерации спецификации**
Спецификация не пишется руками она генерируется Al по документу идеи через промпт-инструкцию. Промпт определяет роль Al, формат вывода, правила и workflow.
* Роль: «Старший продуктовый архитектор, специализирующийся на SaaS>>>
* Контекст: Какие файлы загружены и что содержат
* Задача: «Полная спецификация, покрывающая все модули»
* Формат: Точная структура: разделы, содержимое, уровень детализации
* Правила: Конкретика, полнота, формат user stories, без TODO-заглушек
* Workflow: Прочитай А прочитай в сгенерируй проверь совместимость

**Главное правило**
«Не спрашивай добавляй сам, если ответ есть в исходных документах.» Промпт должен запрещать Al задавать уточняющие вопросы, если ответ уже в файлах.

## 05 Слой 3 — Генератор конфигурации

Генератор документ-инструкция, превращающий спецификацию в набор файлов для Claude Code. Работает как компилятор: на входе спецификация, на выходе файлы для Al-агентов.

**Что генерируется**
* **CLAUDE.md (до 120 строк)**
    Главный конфиг: обзор, стек, архитектура, правила, команды. «README для AI».
* **.claude/agents/*.md (4-6 файлов)**
    Субагенты: database-architect, backend-engineer, frontend-developer, qa-reviewer + опциональные.
* **.claude/rules/ .md (3-5 файлов)**
    Контекстные правила привязанные к файлам через glob-паттерны.
* **.claude/skills/ .md (2-3 файла)**
    Навыки для повторяющихся задач: implement-feature, create-migration.
* **SPEC_TEMPLATE.md**
    Шаблон для описания новых фич: user stories, данные, API, UI, edge cases.
* **МСР-команды**
    Подключение серверов: Context7 (документация), Supabase (БД), GitHub (репозиторий).

**Классификатор проекта**
Перед генерацией анализируется спецификация: есть ли продакшн Al-агенты? Признаки: «Al-ассистенты», «чат-боты», интеграции с LLM, N8n/Make. Если да добавляется субагент ai-agent-architect и соответствующие rules.

**CLAUDE.md Максимум 120 строк**
Claude Code перечитывает его при каждом запуске. Раздутый файл засоряет контекст. Обзор + стек + архитектура + правила + команды и всё.

## 06 Слой 4 — Субагенты

Субагенты специализированные Al-роли внутри Claude Code. Вместо одного агента, который делает всё, работает команда специалистов.

**Стандартная команда**

| Субагент | Модель | Ответственность |
| :--- | :--- | :--- |
| database-architect | opus | Схема БД, миграции, RLS, индексы |
| backend-engineer | opus | API, Server Actions, Edge Functions |
| frontend-developer | sonnet | UI, компоненты, формы, навигация |
| qa-reviewer | sonnet | Code review, тесты RLS, безопасность |
| payments-specialist | opus | Stripe, подписки, вебхуки, биллинг |
| ai-agent-architect | opus | Продакшн Al-агенты для пользователей |

**Выбор модели и инструментов**
* Opus архитектура БД, безопасность, платежи, сложная логика.
* Sonnet UI, простые правки, code review. Sonnet в 2х дешевле экономия заметна на длинных сессиях.
* Важно: qa-reviewer получает только Read, Bash, Glob, Grep без Write и Edit. Ревьюер не меняет код, а описывает проблемы.

**Анатомия промпта субагента**
* YAML-заголовок:
    * пате: английское имя (идентификатор) | description: когда использовать | tools: Read, Write, Edit, Bash, Glob, Grep | model: opus / sonnet
* Тело промпта (на русском):
    1.  Роль кто и в чём специализируется
    2.  Принципы конкретные правила
    3.  Паттерны шаблоны для типичных задач
    4.  Чеклист что проверить перед завершением
    5.  Интеграция МСР и взаимодействие с другими субагентами

## 07 Слой 5 — Промпт-инструкция

Промпт-инструкция управляющий документ, соединяющий все слои. Без него слои существуют по отдельности. С ним каждый следующий автоматически создаётся из предыдущего.

**Три типа промпт-инструкций**
* **Идея - Спецификация**: Загружается вместе с документом идеи. Определяет формат (17 разделов), правила, workflow. Ключевое: «не спрашивай добавляй сам».
* **Спецификация Конфигурация**: Сам файл генератора. 7 частей: классификация, CLAUDE.md, субагенты, rules, skills, шаблоны, МСР. Вывод один промпт для Claude Code.
* **Конфигурация Сборка**: Финальный промпт из генератора. Все файлы с точным содержимым. Копируется в Claude Code тот создаёт структуру автоматически.

**Правила написания промпт-инструкций**
* Роль Al конкретно: не «помощник», а «старший архитектор БД на PostgreSQL и Supabase>>>
* Формат вывода структура документа, порядок разделов, содержимое каждого
* Запрет нежелательного «не используй TODO-заглушки», «не спрашивай если ответ есть»
* Workflow порядок действий, что читать первым, что проверять

**Критерии качества**
«достаточно детально для реализации без уточнений» Язык промпта язык вывода

## 08 Шаблоны и структуры

Все выходные документы copy-paste ready. Без заполнителей, без инструкций «вставьте сюда». Каждый файл копируется целиком в нужное место.

**Шаблон спецификации фичи**
# Спецификация фичи: [НАЗВАНИЕ]
## Описание [Что делает, для кого, зачем]
## User Stories [Как роль, я хочу действие, чтобы результат]
## Модель данных [Таблицы, поля, типы, связи, RLS]
## АРІ [Эндпоинты, Server Actions, Edge Functions]
## Экраны [Страницы, компоненты, состояния]
## Бизнес-логика [Правила, валидации]
## Крайние случаи [Что может пойти не так]
## Приоритет / Зависимости

**Структура репозитория**
* `project-root/`
    * `CLAUDE.md` главный конфиг (до 120 строк)
    * `SPEC TEMPLATE.md` шаблон фичи
    * `.claude/`
        * `agents/` субагенты
        * `rules/` контекстные правила
        * `skills/` навыки
    * `src/` код приложения
    * `supabase/`
        * `migrations/` SQL-миграции
        * `functions/` Edge Functions
    * `package.json`

## 09 Антипаттерны: что убивает Al-разработку

Повторяющиеся ошибки. Каждая стоит часов потерянного времени.
* **Код без спецификации**: Claude Code без контекста генерирует фрагментарный код. Через день полная переделка.
* **Абстрактная спецификация**: «Модуль аналитики с продвинутыми отчётами» Al не знает что значит «продвинутый». Нужны конкретные таблицы, поля, эндпоинты.
* **Раздутый CLAUDE.md**: 300+ строк. Claude Code перечитывает при каждом запуске. Максимум 120 строк. Детали B rules skills.
* **Все субагенты на Opus**: Opus в 2х дороже. UI и code review не требуют Opus. Opus для архитектуры, Sonnet для остального.
* **Ревьюер с правами на запись**: qa-reviewer c Write/Edit начинает «исправлять» вместо описания проблем. Только Read, Bash, Glob, Grep.
* **Плоский дамп файлов**: Всё в корне без структуры. Субагенты не находят нужное. Структура создаётся ДО начала сборки.
* **Без Context7 MCP**: Claude пишет код по устаревшим АРІ. Каждый субагент обязан проверять актуальность через Context7.
* **Ожидание полной автономии**: Claude Code требует одобрения действий. Автономность в качестве решений, а не в отсутствии контроля.

## 10 Чеклист готовности к запуску сборки

Перед вставкой промпта в Claude Code пройди этот чеклист. Каждый пропущенный пункт часы на отладку.

**Документ идеи**
* [ ] Проблема с числами и примерами
* [ ] Решение пошаговый процесс
* [ ] Архитектура с диаграммой слоёв
* [ ] Стек с обоснованием выбора
* [ ] Монетизация с планами и ценами
* [ ] Конкуренты в таблице
* [ ] MVP выделен отдельно от ѵ2

**Спецификация**
* [ ] Каждый модуль: user stories + данные + API + UI + логика + edge cases
* [ ] Типы полей как в SQL
* [ ] АРІ: метод + путь + тело + ответ + коды
* [ ] RLS для каждой таблицы
* [ ] Al-агенты описаны: модель, промпт, tools
* [ ] Нет TODO-заглушек

**Генератор**
* [ ] CLAUDE.md до 120 строк
* [ ] Модели: opus для сложного, sonnet для рутины
* [ ] qa-reviewer без Write/Edit
* [ ] Rules c glob-паттернами
* [ ] MCP: Context7 + Supabase
* [ ] Финальный промпт содержит ВСЕ файлы

**Перед запуском**
* [ ] Репозиторий инициализирован
* [ ] Supabase-проект подключён
* [ ] Переменные окружения настроены
* [ ] Claude Code установлен
* [ ] Файловая структура создана

«Качество спецификации определяет качество автономной сборки. Каждый час в документации экономит день разработки.»
